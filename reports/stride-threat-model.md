## Threats

### Spoofing

#### T-01 — Actor identity spoofing

`MEDIUM likelihood / HIGH impact`

**Asset:** User identity  
**Actor:** External attacker  
**ED-202A:** SO-2: Protect against identity spoofing

An attacker impersonates a legitimate pilot or dispatcher to gain access.

**Mitigations:**
- Implement MFA for pilot login
- Validate token signature on every request

**Suggested test cases:**
- `test_fake_token_rejected`
- `test_expired_token_rejected`

### Tampering

#### T-02 — Flight plan data tampering

`HIGH likelihood / HIGH impact`

**Asset:** Flight plan data  
**Actor:** Malicious insider / compromised account  
**ED-202A:** SO-3: Maintain data integrity

An authenticated user modifies another pilot's flight plan via IDOR.

**Mitigations:**
- Add ownership check before any plan mutation
- Log all write operations with user ID

**Suggested test cases:**
- `test_pilot_cannot_modify_others_plan`
- `test_ownership_enforced_on_delete`

### Repudiation

#### T-03 — Audit log bypass

`MEDIUM likelihood / MEDIUM impact`

**Asset:** Audit trail  
**Actor:** Any authenticated user  
**ED-202A:** SO-4: Support non-repudiation

No request logging means actions cannot be attributed to a specific user.

**Mitigations:**
- Implement structured request logging (user_id, endpoint, timestamp, response_code)
- Store logs in append-only storage

**Suggested test cases:**
- `test_action_logged_on_flight_plan_create`
- `test_auth_attempt_logged`

### Information Disclosure

#### T-04 — Debug endpoint data exposure

`HIGH likelihood / CRITICAL impact`

**Asset:** JWT secret / session tokens  
**Actor:** External attacker  
**ED-202A:** SO-5: Prevent information disclosure

Unauthenticated /debug endpoint exposes JWT secret, active tokens, and env vars.

**Mitigations:**
- Remove /debug endpoint from production
- Move sensitive config to environment vault

**Suggested test cases:**
- `test_debug_returns_403_in_production`
- `test_jwt_secret_not_in_any_response`

### Denial of Service

#### T-05 — Brute-force authentication flood

`HIGH likelihood / HIGH impact`

**Asset:** Authentication service  
**Actor:** External attacker  
**ED-202A:** SO-6: Maintain availability

No rate limiting on /auth/token allows credential stuffing at scale.

**Mitigations:**
- Add rate limiting: max 5 attempts/minute per IP
- Implement progressive delay

**Suggested test cases:**
- `test_rate_limit_returns_429_after_5_attempts`
- `test_lockout_after_threshold`

### Elevation of Privilege

#### T-06 — Role escalation via token replay

`LOW likelihood / HIGH impact`

**Asset:** Role-based access control  
**Actor:** Compromised user account  
**ED-202A:** SO-2: Protect against privilege escalation

Tokens do not encode role at issuance; role is re-fetched from mutable store.

**Mitigations:**
- Embed role in signed token payload
- Invalidate tokens on role change

**Suggested test cases:**
- `test_role_change_invalidates_existing_token`
- `test_pilot_token_rejected_on_maintenance_route`

## Attack Trees

### T-02 — Read or modify another pilot's flight plan

```
Access any flight plan via IDOR
  └─ Obtain valid Bearer token (legitimate login or credential stuffing)
  └─ Enumerate plan IDs (sequential: fp001, fp002...)
    └─ Use /debug to get all plan IDs without auth
  └─ GET /flightplans/<id> — no ownership check → 200
```
